# 方案二：基于代码知识库的仓库问答设计

## 背景

当前系统已经具备两类能力：

1. 提交 GitHub 仓库后生成中文分析报告
2. 任务成功后，在任务详情页继续追问

但是现有问答能力仍然主要依赖任务分析结果和少量文件片段，无法稳定深入理解真实代码实现。用户已经明确反馈，方案一的回答“过于浅显”，原因不是聊天入口，而是问答前的上下文准备不足。系统要想真正像一个“懂代码的仓库学习 Agent”，必须先建立代码知识库，再让模型基于知识库回答。

## 用户确认的方案二约束

- 知识层存储：`SQLite + 本地文件切片索引`
- 索引范围：只索引源码与配置文件，排除构建产物和大文件
- 回答方式：既要交给大模型分析，也要把命中的代码片段和文件位置展示给用户
- 索引建立时机：任务分析成功后自动执行“知识建库”阶段，建完后才能问答

## 目标

把当前“报告驱动问答”升级为“代码知识库驱动问答”，使系统能够：

- 在任务成功后自动建立可检索的代码知识库
- 在用户提问前先检索真实代码，而不是只看 Markdown 报告
- 让大模型基于命中的源码片段给出中文结论
- 同时把文件路径、行号和代码片段原文展示给用户
- 在模型不可用时，至少仍然返回命中的代码依据

## 总体架构

### 当前链路

当前链路在任务成功时结束于：

1. 仓库分析完成
2. 报告生成成功
3. 任务状态置为 `succeeded`

### 方案二链路

方案二将任务成功后的流程延长为：

1. 仓库分析完成
2. 报告生成成功
3. 自动进入知识建库阶段
4. 切片源码与配置文件
5. 将切片写入 SQLite 知识库
6. 建立全文索引
7. 知识库状态变为可用
8. 前端开放问答输入
9. 用户提问
10. 后端先检索代码片段
11. 大模型基于检索结果生成中文回答
12. 前端同时展示“中文结论 + 文件位置 + 代码片段”

也就是说，问答前的前置条件不再只是“任务成功”，而是“任务成功且知识库已就绪”。

## 数据存储设计

### 存储介质

每个任务单独维护一个本地 SQLite 文件，建议路径：

`artifacts/<task_id>/knowledge.db`

这样有几个好处：

- 与当前任务产物天然绑定
- 生命周期容易和任务产物一起管理
- 不需要单独引入数据库服务
- 未来可以平滑迁移到更强的检索方案

### 数据表设计

#### 1. `task_knowledge_document`

记录纳入知识库的文件。

字段建议：

- `id`
- `task_id`
- `path`
- `file_type`
- `language`
- `size_bytes`
- `is_indexed`
- `created_at`

职责：

- 保存任务内已扫描文件的元信息
- 标记哪些文件被索引，哪些文件被过滤掉

#### 2. `task_knowledge_chunk`

记录切片后的代码块。

字段建议：

- `id`
- `task_id`
- `document_id`
- `chunk_index`
- `path`
- `start_line`
- `end_line`
- `symbol_name`
- `chunk_kind`
- `content`
- `summary`
- `token_estimate`

职责：

- 保存检索的最小工作单元
- 每个 chunk 必须能反向定位到文件和行号

#### 3. `task_knowledge_fts`

SQLite FTS5 虚表，用于全文检索。

索引字段建议：

- `path`
- `symbol_name`
- `summary`
- `content`

职责：

- 负责首轮召回

#### 4. `task_chat_session`

字段建议：

- `id`
- `task_id`
- `created_at`
- `updated_at`

职责：

- 保存任务级问答会话

#### 5. `task_chat_message`

字段建议：

- `id`
- `session_id`
- `role`
- `content`
- `answer_summary`
- `confidence`
- `created_at`

职责：

- 保存用户问题和 Agent 回答

#### 6. `task_chat_citation`

字段建议：

- `id`
- `message_id`
- `chunk_id`
- `path`
- `start_line`
- `end_line`
- `reason`
- `snippet`
- `rank`

职责：

- 保存回答与命中代码片段之间的关系

### 预留扩展位

虽然第一版不引入向量检索，但建议在 `task_knowledge_chunk` 层为未来预留：

- `embedding_provider`
- `embedding_model`
- `embedding_vector_ref`

这样后续如果要升级为混合检索，不需要重设计主表。

## 索引范围与过滤规则

### 纳入索引

- Python 源码
- TypeScript / JavaScript 源码
- Vue / React 组件文件
- YAML / TOML / JSON 配置
- Docker / Compose / CI / K8s 相关配置
- 环境模板文件

### 不纳入索引

- `.git`
- `node_modules`
- `dist`
- `build`
- 二进制文件
- 图片资源
- 体积超限文件
- 当前任务生成的 Markdown / HTML / PDF 报告

这里明确不把报告作为知识库主体，因为用户要求的是“分析项目中的代码”，不是继续围绕报告改写。

## 切片策略

### 切片目标

切片不是机械按固定字符数拆分，而是尽量让每个片段具备“可解释的代码边界”。

### 第一版建议

采用“结构优先，长度兜底”的混合切片：

1. 优先按符号边界切片
   - Python：函数、类、路由定义附近
   - 前端：组件、hooks、store、API 调用附近
   - 配置：service、job、env block、resource block
2. 如果无法识别结构，则按行数切片
   - 例如每片 30 到 80 行
3. 相邻切片保留少量重叠
   - 例如 5 到 10 行

### 切片附加信息

每个 chunk 附带：

- 文件路径
- 起止行
- 语言
- 结构类型
- 可选符号名
- 轻量摘要

这样即使模型不可用，系统也能直接把命中的片段给用户看。

## 建库时机与任务状态

### 新增阶段

建议在任务阶段枚举中新增：

- `build_knowledge`

### 状态行为

主分析流程完成后，不立即将任务标记为最终可问答，而是进入：

- `running / build_knowledge / 95%`

建库成功后：

- `succeeded / finalize / 100%`
- 附加 `knowledge_ready = true`

建库失败时：

- 任务仍可保留分析结果
- 但问答入口禁用
- 页面明确显示“知识库构建失败”

### 为什么不把建库放到第一次提问时

因为用户已经明确选择“分析成功后自动建库”。这样体验更稳定：

- 用户打开结果页时就知道是否可问答
- 不会第一次提问时才触发较长等待
- 后端可在任务完成后一次性准备好后续上下文

## 检索流程设计

### 目标

让模型看到的不是“猜测性摘要”，而是真实命中的代码片段。

### 流程

#### 1. 问题预处理

对用户问题做轻量解析，提取：

- 路径词
- 函数词
- 路由词
- 组件词
- 报错词
- 配置词

#### 2. 全文召回

使用 SQLite FTS5 按以下字段召回：

- `content`
- `summary`
- `path`
- `symbol_name`

#### 3. 规则重排

对召回结果做二次排序，按以下因子加权：

- 路径直接命中
- 文件名直接命中
- 符号名命中
- 问题类型与文件类型匹配
- 关键词密度
- 是否为入口文件
- 是否属于前后端调用链关键文件

#### 4. 选取上下文

取前 N 个 chunk 作为模型输入上下文，例如：

- 4 到 8 个 chunk
- 每个 chunk 限制在一定行数

#### 5. 模型生成

大模型必须基于这些 chunk 回答：

- 输出中文结论
- 标出仓库事实
- 单列补充说明
- 不允许脱离证据乱答

#### 6. 前端展示

最终页面展示：

- 中文解释
- 引用文件路径
- 起止行号
- 原始代码片段

## 模型提示词设计

### 硬性要求

- 必须使用简体中文
- 优先使用检索到的代码片段作答
- 不得伪造代码实现
- 所有判断尽量绑定到具体文件和行号
- 如果某条结论是推断，必须显式说“这是推断”
- 如果证据不足，必须明确说明证据不足

### 输出结构

建议模型输出 JSON：

- `answer`
- `citations`
- `supplemental_notes`
- `confidence`

其中 `citations` 必须与检索到的 chunk 对齐，不允许模型随意编造不存在的行号。

## 前端展示设计

### 页面结构

任务详情页仍保持“报告 + 聊天”双栏，但聊天区升级为“知识库问答模式”。

### 新增状态

- `知识建库中`
- `知识库已就绪`
- `知识库构建失败`

### 回答展示

每条回答由两部分组成：

#### 1. 中文结论区

- Agent 的解释性回答
- 补充说明
- 置信度

#### 2. 代码依据区

逐条展示：

- 文件路径
- 起止行号
- 代码片段原文
- 命中原因

这样用户既能看懂，也能自己回到源码核实。

## 降级策略

### 模型不可用

如果大模型不可用：

- 仍然返回命中的代码片段
- 由系统生成较短的规则化解释
- 不让用户空手而归

### 检索为空

如果知识库检索不到足够证据：

- 明确告诉用户没有找到足够直接的代码证据
- 返回最接近的文件候选或入口文件建议

### 知识库损坏

如果 SQLite 文件损坏或缺失：

- 标记任务知识库不可用
- 前端显示错误
- 可提供“重新建库”能力作为后续增强

## 测试要求

### 后端

- 任务成功后自动建库
- SQLite 文件成功生成
- chunk 写入与 FTS 索引写入正确
- 问答接口能返回真实代码片段
- 回答包含路径与行号
- 模型失败时仍返回代码片段

### 前端

- 知识建库中状态展示
- 知识库就绪后开放提问
- 回答区同时展示结论与代码依据
- 代码依据展示路径与行号
- 知识库失败时正确提示

## 实施建议

建议按两个子阶段落地：

### 第一子阶段

- 增加 `build_knowledge` 阶段
- 增加 SQLite 建库
- 实现 FTS5 检索
- 问答接口改接知识库

### 第二子阶段

- 提升切片质量
- 增加结构化重排
- 增加重新建库和检索调试信息

## 结论

方案二的本质，不是“把聊天功能做重”，而是把系统从“报告生成器 + 浅层问答”升级为“基于真实代码知识库的学习 Agent”。

当前推荐直接按本设计执行：

- 任务成功后自动建库
- 用 SQLite + FTS5 做第一版检索
- 让大模型基于真实代码片段回答
- 同时把代码片段和文件位置展示给用户

这样能显著提升回答深度，并且后续还可以继续演进到更强的混合检索能力。
