# 方案三：基于仓库认知图的 Agent 问答设计

## 背景

方案二已经把任务级问答从“报告驱动”升级为“代码知识库驱动”：

- 任务成功后自动构建 `knowledge.db`
- 只索引源码与配置文件
- 问答时返回中文结论、文件路径、行号和真实代码片段

但方案二的核心能力仍然是“切片检索 + 局部总结”。它能较好回答“哪个文件”“哪段代码”“哪个接口”，却不擅长回答以下问题：

- 项目里是否实际存在某种系统能力
- 某个模块在整体架构里扮演什么角色
- 前端请求如何跨文件流转到后端
- 一个概念是否只是字符串出现，还是已经形成真实实现

原因不是单纯模型能力不足，而是当前系统没有“仓库级认知层”。它没有在问答前建立完整的项目地图，也不会围绕问题主动做多跳调用链推理。

## 用户确认的方案三约束

- 先做“仓库认知图 / 项目地图”，再增强问答
- 首版优先覆盖“代码结构 + 调用链”
- 认知图要做成结构化图谱，而不是只保留文件级摘要
- 问答仍然必须落回真实代码证据，不允许只给抽象推断

## 目标

把当前系统从“代码切片检索器”升级为“具备仓库整体理解能力的学习 Agent”，使系统能够：

- 在任务完成后自动构建仓库认知图
- 识别文件、符号、路由、组件、API 调用与 import 关系
- 形成前后端调用链与关键入口图
- 在用户提问时，先利用认知图做问题规划和多跳定位
- 再结合真实代码片段生成中文回答
- 明确区分“直接证据”和“基于图谱的推断”

## 总体思路

### 现有方案二链路

1. 分析仓库
2. 生成报告
3. 构建 `knowledge.db`
4. 检索代码切片
5. 大模型或本地兜底生成回答

### 方案三增强后的链路

1. 分析仓库
2. 生成报告
3. 构建 `knowledge.db`
4. 构建 `repo_map.json`
5. 用户提问
6. 问题规划器先判断问题类型
7. 从认知图中召回入口节点、关系边、调用链摘要
8. 再回到 `knowledge.db` 补齐代码证据
9. 大模型基于“认知图结论 + 代码证据”生成中文回答
10. 前端展示：
   - 结论
   - 回答来源
   - 认知图命中信息
   - 代码证据

## 首版范围

首版只做“代码结构 + 调用链优先”，不做业务语义抽取。

### 纳入认知图的对象

- 文件节点
- 目录到文件的归属
- Python 函数 / 类
- 前端组件 / hook / store
- 后端路由
- 前端 API 调用
- import 关系
- 前后端映射边
- 关键入口文件

### 暂不纳入首版

- 通用业务概念抽取
- 向量索引
- 动态运行时追踪
- 自主长链路规划 Agent
- 跨任务共享的全局知识图谱

## 认知图数据模型

认知图以任务级 JSON 文件持久化，建议路径：

`artifacts/<task_id>/repo_map.json`

首版使用 JSON 而非额外数据库，原因：

- 与当前任务产物生命周期一致
- 易于调试和人工检查
- 先验证结构和问答价值，再决定是否迁移到 SQLite 表

### 1. 文件节点 `file_nodes`

字段建议：

- `id`
- `path`
- `kind`
- `language`
- `role`
- `exports`
- `imports`
- `summary`

职责：

- 形成仓库的文件级地图
- 标记入口文件、配置文件、普通实现文件

### 2. 符号节点 `symbol_nodes`

字段建议：

- `id`
- `file_path`
- `name`
- `kind`
- `start_line`
- `end_line`
- `signature`
- `summary`

职责：

- 表达“这个文件里定义了什么”
- 为问答提供比文件更细粒度的认知对象

### 3. 关系边 `edges`

字段建议：

- `source_id`
- `target_id`
- `type`
- `evidence`
- `confidence`

首版关系类型：

- `imports`
- `defines`
- `calls_api`
- `defines_route`
- `maps_to_backend`
- `uses_config`
- `entrypoint_of`

职责：

- 让系统能跨文件理解结构，而不是只看单文件片段

### 4. 入口摘要 `entrypoints`

字段建议：

- `backend`
- `frontend`
- `config`

每项至少包含：

- `file_path`
- `reason`
- `related_symbols`

职责：

- 快速回答“入口在哪”“从哪看起”

### 5. 调用链摘要 `call_chains`

字段建议：

- `id`
- `name`
- `question_types`
- `steps`
- `confidence`

每个 `step` 至少包含：

- `kind`
- `path`
- `symbol`
- `note`

职责：

- 把静态关系压缩成问答可消费的高价值链路
- 例如“前端 fetch -> 后端 route -> handler”

## 构建策略

### 文件层

沿用方案二已筛出的源码与配置文件，只在这些文件上构建认知图，避免重复扩展范围。

### 结构识别

首版使用静态规则优先：

- Python：`def` / `class` / FastAPI 路由装饰器
- Vue / React：组件导出、`fetch` / `axios` 调用、常见 store/hook 命名
- 配置：`docker-compose.yml`、`.env`、`k8s/*.yml`、`pyproject.toml`、`package.json`

### 调用链识别

首版只做高置信规则：

- 前端 API URL 与后端 route path 直接匹配
- 入口文件 import 下游文件
- 后端路由函数落在同文件或同模块中

如果证据不足，不强行连边。

## 问答增强设计

### 问题分类

先将问题归类为：

- 入口定位
- 模块职责
- 调用链
- 是否存在某能力
- 配置 / 部署关系
- 具体实现细节

### 检索顺序

方案三不再只查 chunk，而是采用：

1. 查 `repo_map.json`
2. 找相关文件、符号、边、调用链
3. 生成结构化上下文
4. 再查 `knowledge.db` 取代码证据
5. 组合成最终 prompt

### 回答结构

回答必须区分两层：

1. `仓库结论`
2. `代码证据`

并附加：

- `answer_source`
- `graph_evidence`
- `citations`
- `supplemental_notes`
- `confidence`

其中：

- `graph_evidence` 用于告诉用户“这条结论基于哪些节点和边”
- `citations` 仍然只引用真实代码片段

## 前端展示设计

任务详情页聊天区继续保留，但每条回答增加两类可视信息：

### 1. 认知图结论

- 告诉用户系统是从哪条结构关系得出结论
- 例如：`前端组件 -> API 调用 -> 后端路由`

### 2. 代码证据

- 文件路径
- 行号
- 原始代码片段

这样用户能看出：

- 系统不是只做字符串命中
- 结论是如何落到代码上的

## 错误与降级

### 认知图构建失败

- 保留方案二的 `knowledge.db`
- 任务仍然可问答
- 但问答退回方案二模式
- 前端显示“认知图未就绪，当前回答基于代码知识库”

### 大模型不可用

- 仍然返回认知图命中与代码证据
- 本地回答来源标记为 `local`

### 图谱证据不足

- 明确说“当前认知图中没有足够证据确认”
- 不允许把弱推断伪装成确定事实

## 测试要求

### 后端

- 认知图构建成功
- 文件/符号/边提取正确
- 调用链摘要正确
- 问答先查图再查代码
- 认知图失败时回退到方案二
- 大模型失败时仍返回图证据和代码证据

### 前端

- 正确展示认知图命中信息
- 正确展示回答来源
- 认知图未就绪时展示降级状态
- 代码证据渲染不受影响

## 分阶段落地建议

### 第一阶段

- 增加 `repo_map.json`
- 增加图构建器
- 增加问题分类器
- 增强聊天服务优先使用认知图
- 前端展示回答来源与认知图命中

### 第二阶段

- 提升边关系质量
- 增加更细粒度调用链
- 增加“推荐追问”
- 增加调试面板

## 结论

方案三的本质不是“换个更大的模型”，而是给系统新增一层仓库级认知能力。  
它要解决的是“系统能不能理解整个项目”这个问题，而不是仅仅让模型在更多代码片段上做文本总结。

首版最合理的落地方式是：

- 保留方案二的 `knowledge.db`
- 增加任务级 `repo_map.json`
- 问答时先查认知图，再补代码证据
- 让前端明确展示“认知图结论 + 代码证据”
