# 仓库报告与持续问答 Agent 设计

## 背景

当前系统已经具备“提交 GitHub 仓库 -> 异步分析 -> 生成中文报告”的能力，但用户在阅读报告时遇到不理解的实现细节、调用链路或排错问题时，无法继续基于该仓库追问。系统形态更接近一次性报告生成器，而不是持续陪伴式的仓库学习 Agent。

本设计的目标是把任务详情页升级为“报告 + 问答工作台”。用户在分析任务成功后，不仅能查看报告，还能继续围绕当前仓库提问，并获得带代码依据的中文回答。

## 用户确认的核心约束

- 提问入口固定放在任务详情页，与报告并排展示
- 回答以当前仓库和已生成报告为主，允许补充少量通用开发知识
- 每次回答都必须附带代码依据
- 只有任务分析成功后才允许开始问答
- 所有 AI 输出必须严格使用中文

## 目标

### 方案一目标

快速在现有任务体系上增加可用的仓库问答能力，让用户在分析完成后可直接追问。

### 方案二目标

在方案一验证价值后，进一步把仓库问答能力升级为独立知识层，提升检索精度、多轮体验和后续扩展性。

## 方案对比

### 方案一：直接复用现有任务产物做问答

#### 核心思路

分析任务成功后，后端直接复用当前任务已有的：

- `AnalysisResult`
- `repo_path`
- 扫描得到的文件列表
- 已生成的关键代码片段和教程摘要

当用户提问时，后端先在当前仓库内做轻量检索，再把“问题 + 任务摘要 + 命中文件片段”发给大模型，生成严格中文回答。

#### 优点

- 改动小，能够快速上线
- 与现有任务模型天然对齐，不需要新建独立知识库
- 可以直接使用已经保存的任务访问控制和任务详情页上下文

#### 缺点

- 检索效果依赖首版策略，复杂问题的召回能力有限
- 多轮会话的上下文压缩和长期记忆能力较弱
- 后续高级能力需要继续演进

### 方案二：增加独立的仓库知识库层

#### 核心思路

分析完成后，额外执行一层“知识准备”：

- 对仓库文件切片
- 提取结构化元数据
- 生成可检索的索引
- 持久化问答会话和检索候选

问答时不再临时扫描仓库，而是先查知识层，再把更高质量的命中结果交给模型生成答案。

#### 优点

- 检索质量更高，复杂追问更稳
- 更适合多轮问答、会话记忆、后续推荐问题、相似问题缓存
- 后续可扩展到跨任务知识复用、管理台检索、知识回放

#### 缺点

- 实现成本明显更高
- 涉及索引生命周期、切片策略、存储结构和额外测试面
- 第一版如果直接上，会拖慢交付和验证速度

## 推荐落地路径

### 第一阶段

先做方案一，目标是验证下面三个问题：

1. 用户是否真的会在报告阅读过程中持续提问
2. 当前基于任务产物的检索质量是否足够支撑常见问题
3. 用户是否认可“带代码依据的中文回答”这种交互形态

### 第二阶段

如果方案一使用反馈表明：

- 用户提问频率高
- 问题复杂度明显超出第一版检索能力
- 用户对定位精度和多轮连续性要求更高

再启动方案二，把问答能力从“任务附属能力”升级为“仓库知识能力”。

## 方案一详细设计

### 前端

任务详情页拆成左右双栏：

- 左侧保留现有任务状态、时间线和分析报告
- 右侧新增固定聊天区

聊天区能力：

- 展示欢迎语和提问限制
- 用户输入问题并发送
- 展示提问中状态
- 展示回答内容
- 展示引用文件及代码片段
- 区分“仓库依据”和“通用补充”

聊天区只在任务 `succeeded` 时可用；任务未完成、失败或取消时隐藏或禁用。

### 后端

新增任务级问答接口：

- `POST /api/v1/tasks/{task_id}/chat`
- `GET /api/v1/tasks/{task_id}/chat/messages`

接口继续沿用任务访问控制，允许通过任务 token 或具备足够 scope 的全局凭证访问。

问答只允许针对成功任务执行。失败、取消、运行中任务请求问答时返回明确错误。

### 检索层

方案一不引入向量库，采用轻量检索：

1. 从 `AnalysisResult` 中抽取高价值摘要：
   - 后端路由
   - 前端路由与组件
   - API 调用
   - 逻辑流
   - 教程代码走读
2. 根据问题做关键词归一化
3. 在摘要中筛选候选文件
4. 从仓库文件中抽取有限数量的片段
5. 将候选片段按相关性排序后送入模型

这样可以控制实现成本，同时保留后续升级到方案二的接口边界。

### 提示词策略

提示词必须严格限制：

- 必须使用简体中文
- 优先回答仓库中能直接验证的事实
- 允许补充少量通用知识，但必须单独标注为“补充说明”
- 不能编造仓库中不存在的实现
- 每次回答都必须给出引用文件
- 如果证据不足，必须明确说明“当前仓库中没有足够证据”

模型返回结构化 JSON，至少包含：

- `answer`
- `citations`
- `supplemental_notes`
- `confidence`

### 数据模型

方案一新增两类核心数据：

1. 任务级聊天消息
   - 用户问题
   - Agent 回答
   - 时间戳
2. 回答引用
   - 文件路径
   - 行号范围或片段文本
   - 引用原因

建议仍然保存在 Redis 中，以便快速接入并与当前任务生命周期保持一致。

## 方案二详细设计

### 目标形态

方案二将新增“仓库知识准备”子系统，位于分析任务之后、问答接口之前。

### 新增能力

- 文件切片与分段
- 元数据索引
- 语义检索或混合检索
- 会话级上下文压缩
- 推荐追问
- 常见问题缓存

### 数据层

至少新增以下实体：

- `task_knowledge_index`
- `task_knowledge_chunk`
- `task_chat_session`
- `task_chat_message`
- `task_chat_retrieval_trace`

### 检索流程

1. 任务成功后建立索引
2. 用户提问时先查索引
3. 召回 chunk
4. 重排
5. 把高相关 chunk 与问题一起送给模型
6. 记录检索轨迹，便于调试和优化

### 方案二价值

方案二不是替代方案一，而是建立在方案一之上的升级路径。方案一先验证交互价值，方案二再解决检索精度和多轮质量问题。

## 错误处理

无论方案一还是方案二，都要统一处理以下情况：

- 任务不存在
- 任务未完成
- 任务已取消或失败
- 仓库产物已被清理
- 大模型不可用
- 检索不到足够依据

在大模型不可用时，应尽量返回基于已有结构化分析结果的降级回答，而不是直接报错。

## 测试要求

### 后端

- 成功任务问答
- 运行中任务问答被拒绝
- 失败任务问答被拒绝
- 大模型回答结构校验
- 中文输出校验
- 引用文件必填校验
- 降级回答路径测试

### 前端

- 成功任务展示聊天区
- 非成功任务隐藏或禁用聊天区
- 发送问题后渲染答案
- 渲染引用文件与片段
- 错误状态展示

## 结论

当前建议：

- 立即实现方案一，尽快让用户可体验“报告 + 仓库问答”
- 将方案二作为明确记录的升级路线保留
- 如果方案一反馈不佳或精度不足，再启动方案二
