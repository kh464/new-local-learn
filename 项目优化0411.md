  第一层
  仓库接入与预处理层

  用户提交 GitHub 链接后，系统第一层先做基础准备，不做回答。

  这一层做的事：

  - 克隆仓库到本地工作目录
  - 扫描文件树
  - 识别语言类型
      - Python
      - Java
      - C/C++
      - 前端文件也可顺带识别
  - 过滤无价值目录
      - .git
      - node_modules
      - dist
      - build
      - 二进制文件
  - 建立仓库基础清单
      - 文件列表
      - 目录树
      - 语言分布
      - 入口文件候选

  这一层的产物不是给用户看的报告，而是给后续理解层用的“原始材料”。

  第二层
  代码理解与结构抽取层

  这一层开始真正“理解仓库”，但这里的“理解”不是让 LLM 直接通读整个仓库，而是先做结构化抽取。

  这一层会为每个代码文件提取：

  - 模块名
  - 类名
  - 方法名
  - 函数名
  - import / include / using 关系
  - 路由定义
  - 配置定义
  - 继承关系
  - 调用关系候选

  如果先以 Python 为第一站，这一层可以重点抽：

  - 文件级 module
  - class
  - function / method
  - import
  - decorator 路由
  - function call

  这一层产出的是“符号表”和“关系边”。

  也就是：

  - 节点
      - file
      - class
      - function
      - method
      - route
      - config
  - 边
      - contains
      - imports
      - calls
      - inherits
      - routes_to

  这就是你说的“代码神经网络图”的底座。

  第三层
  中文描述生成层

  这是你非常需要的一层，也是混合 RAG 的关键。

  系统不再只是把原始代码切 chunk 存起来，而是为不同粒度对象生成中文描述：

  - 文件描述
      - 这个文件总体负责什么
  - 类描述
      - 这个类的职责是什么
  - 方法描述
      - 这个方法的输入、输出、核心行为是什么
  - 模块描述
      - 这个模块在整个仓库中承担什么角色
  - 调用链描述
      - 某个入口一路调用到哪些关键逻辑

  注意，这一层的描述不是最终回答用户，而是“知识索引材料”。

  也就是说，后面用户提问时，系统先查这些中文描述，而不是先瞎搜原始代码文本。

  这就是从“代码块检索”升级到“代码语义对象检索”的核心变化。

  第四层
  混合 RAG 存储层

  这里就是你问的重点：原来 SQLite 怎么改。

  原来的 SQLite 主要做的是：

  - 文档表
  - chunk 表
  - FTS5 全文索引
  - BM25 排序

  升级后，不建议让 SQLite 单独承担主检索任务，而应该拆成两部分：

    1. 结构化/图谱存储

  - 可以继续用 SQLite
  - 存节点、边、路径、符号元数据
  - 优点是简单、稳定、易落地

    2. 向量检索存储

  - 新增向量数据库或向量索引层
  - 存“文件描述 / 类描述 / 方法描述 / 模块描述 / 调用链描述”的 embedding

  所以“更换为混合 RAG”更准确地说是：

  - SQLite 不再是主召回引擎
  - SQLite 退到元数据和图谱存储层
  - 语义召回交给向量索引
  - 精确召回仍靠结构化索引和必要的全文检索

  这层最终会有三类数据：

  - 元数据
      - 文件路径、语言、symbol、行号
  - 图谱数据
      - 节点和边
  - 向量数据
      - 中文描述 embedding

  第五层
  问题理解与检索规划层

  这是用户提问后触发的第一层运行时逻辑。

  当用户问：

  - “这个项目有知识库吗”
  - “登录流程是怎么走的”
  - “后端收到前端请求后做了什么”
  - “逐行解释 main.py”

  系统不应该直接先搜代码。

  正确顺序应该是：

    1. 先让 LLM 分析用户问题

  - 用户到底问的是能力是否存在
  - 还是调用链
  - 还是模块职责
  - 还是逐行解释
  - 还是配置部署

    2. 生成稳定的检索目标
     例如把：

  - “项目中有知识库吗”
  - “仓库是否实现知识库能力”
  - “是否支持代码索引问答”

  统一收敛成一个内部目标：

  - “确认仓库是否实现知识库/检索问答能力，并定位核心实现模块”

  这一步仍然要用 LLM，因为这是语义理解最强的地方。

  但从这里开始，LLM 不直接找答案，而是交给检索系统去找证据。

  第六层
  混合召回层

  这是新架构和旧架构最大的差别。

  旧架构大致是：

  - 用户问题
  - planner 产出 search query
  - 用关键词搜索 chunk
  - 交给 LLM 回答

  新架构改成：

    1. 精确召回
     先查：

  - 文件名
  - 类名
  - 方法名
  - 路由名
  - 配置名

  如果用户问题里出现显式标识符，这一步优先命中。

    2. 向量召回
     拿用户问题去搜：

  - 文件中文描述向量
  - 类描述向量
  - 方法描述向量
  - 调用链描述向量

  得到语义相关节点。

    3. 图谱扩展
     以召回到的节点为中心，沿图扩展：

  - 上游调用者
  - 下游被调用者
  - 所在文件
  - 所属类
  - 路由入口
  - 相关配置

    4. 代码定位
     把这些节点反查成真实代码位置：

  - 哪个文件
  - 哪几行
  - 哪个 symbol

  这一步之后，系统拿到的已经不是模糊 chunk，而是一组“相关节点 + 真实代码证据”。

  第七层
  证据装配层

  这一层负责把杂乱的检索结果装配成 LLM 真正能吃的上下文。

  会组装成四类证据：

  - 结构证据
      - 入口文件
      - 关键模块
      - 关键类/函数
  - 图谱证据
      - 调用链
      - 依赖关系
      - 所属关系
  - 代码证据
      - 真实代码片段
      - 文件路径
      - 行号范围
  - 描述证据
      - 文件/类/方法中文描述

  然后会做裁剪，避免把整个仓库都塞给 LLM。

  这一层实际上相当于 Codex/Claude Code 的“收集上下文”阶段。

  第八层
  最终回答层

  最后一层必须还是 LLM。

  但这个 LLM 的职责已经变了。

  它不是：

  - 自己猜仓库结构
  - 自己凭印象补逻辑
  - 看几个关键词就瞎说

  它应该只做两件事：

    1. 基于证据推理

  - 根据结构证据 + 图谱证据 + 代码证据组织答案

    2. 用中文表达

  - 给出清晰解释
  - 必须引用文件路径/函数/调用链
  - 证据不足就明确说不足

  也就是说，最终回答仍然是大模型生成，但前面的检索和证据收集不再主要依赖大模型自由发挥。

  这就是你想要的“把 Codex 那种能力赋给系统”的关键设计：
  不是让模型更会猜，而是让系统先把证据找全，再让模型解释。

  新的完整链路

  用户提交仓库链接时：

    1. 克隆仓库
    2. 扫描文件树
    3. 抽取文件/类/方法/调用关系
    4. 生成中文描述
    5. 建立代码图谱
    6. 建立向量索引
    7. 保存元数据和图谱
    8. 生成仓库级报告

  用户后续提问时：

    1. LLM 理解问题意图
    2. 生成稳定检索目标
    3. 精确召回 symbol / file / route
    4. 向量召回相关描述节点
    5. 按图谱扩展上下文
    6. 打开真实代码片段
    7. 组装证据包
    8. LLM 基于证据生成中文回答

  和旧方案相比，本质提升是什么

  旧方案的问题是：

  - 主要靠关键词检索 chunk
  - 用户换个说法，召回就漂
  - 系统很像“搜代码片段”
  - 不像“理解仓库”

  - 先理解代码对象，再做检索
  - 先找相关节点，再找相关代码
  - 用户自然语言可以先命中“中文描述”
  - 再通过图谱拿到完整上下文
  - 最终回答更像“在理解项目之后解释给用户听”

  我给你的推荐落地顺序

  如果开始做，顺序不要乱：

    1. 先定义统一图模型
    2. 先做 Python 适配器
    3. 先生成文件/类/方法中文描述
    4. 先给这些描述做 embedding
    5. 先做混合召回
      - 精确索引 + 向量检索
    6. 再接图谱扩展
    7. 最后再替换现有问答链路

  方法摘要：输入输出、核心逻辑、调用下游、副作用。
  模块摘要：这个目录/模块在整体系统中的位置。

  只能中文。
  只能基于输入证据。
  不能编造。

  你说的“代码神经网络图”，本质上就是：
  节点 = 文件 / 类 / 方法 / 路由 / 配置 / 表
  边 = 调用 / 导入 / 继承 / 映射 / 依赖 / 事件流

  中文描述升级为 LLM 深度理解生成，答案是可以，而且应该做
  我建议不要直接全量一次性让 LLM 读整个仓库，那样成本高且不稳定。
  正确做法是分层生成：

    1. 本地解析代码，抽 AST 和图谱事实。
    2. 给每个文件/符号构造结构化证据包。
    3. 调用 LLM 生成中文摘要。
    4. 摘要落库，带版本号、内容哈希、模型名。
    5. 仓库变化后只增量重算变更节点。

  这样做的优点：
  成本可控。
  摘要可缓存。
  可增量更新。
  更适合大仓库。
  稳定性比“整仓一次喂给大模型”高很多。

    1. 先做 LLM 文件/符号摘要生成链路。
    2. 再做 TypeScript/Java 适配器，补多语言图谱。
    3. 再做“认知层检索 -> 图谱扩展 -> 代码证据定位”主问答链路。
